## 语言与表达

- 所有回答、文档和新增说明均使用中文。
- 输出结论前先确认上下文，避免只根据局部文件或单条命令结果下判断。
- 审核或总结时优先给出高信号结论，必要时再补充验证过程。

## 文件读取与搜索

- 搜索文件和文本时优先使用 `rg`。
- 在 PowerShell 中读取中文 Markdown 时，优先显式指定 UTF-8：

```powershell
Get-Content -Encoding UTF8 path\to\file.md
```

- 如果命令输出中文乱码，不要直接判断文件内容损坏，应先确认读取编码。
- 查看文档时先看目录结构和标题大纲，再阅读重点段落。

## OpenSpec 操作

- 在 Windows PowerShell 中运行 OpenSpec 命令时，优先使用 `.cmd` 入口：

```powershell
openspec.cmd validate <change-id> --strict
```

- 结构校验通过只代表格式有效，不代表内容设计已经合理；仍需人工检查一致性、可执行性和测试覆盖。
- 审核 OpenSpec 变更时，优先交叉阅读 proposal、design、tasks 和 specs，确认它们互相对齐。

## 审核思路

- 关注文档之间是否互相矛盾。
- 关注设计是否有完整执行路径。
- 关注任务列表是否覆盖关键实现和验证工作。
- 关注安全、隐私、失败处理和边界条件是否被明确说明。
- 对不确定点标为待澄清，不要把推测写成事实。

## Git 与工作区

- 修改前后查看工作区状态，确认只产生当前任务需要的变更。
- 不要回滚、删除或改写与当前任务无关的已有变更。
- 遇到与当前任务无关的 warning，先判断是否影响工作，不要擅自修改用户环境或全局配置。
